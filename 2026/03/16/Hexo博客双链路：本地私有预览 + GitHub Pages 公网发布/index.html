Hexo 博客双链路：本地私有预览 + GitHub Pages 公网发布

发表于2026-03-16|更新于2026-03-16|博客搭建
|浏览量:

为什么要把博客拆成两条链路

写博客一段时间以后，很容易出现两类内容：

  • 一类适合公开发布，比如博客搭建、工具使用、音乐和观影记录
  • 一类更适合只在本地整理，比如专题学习笔记、还没写完的草稿、暂时不想公开的长文系列

如果这两类内容全都混在同一个 source/_posts/ 里，后面会有两个问题：

  • 本地预览时太杂，找不到真正要写的私有内容
  • 一旦直接执行 npm run deploy，就有可能把本地私有内容一起发到 GitHub Pages

所以更稳的做法是把博客拆成两条链路：

  • 公网链路：只负责正式公开内容
  • 本地链路：只负责私有文章的本地生成和预览，不参与公网部署

这套项目现在就是按这个思路组织的。

当前项目里的两条链路分别是什么

1. 公网链路

公网链路继续使用 Hexo 的标准目录和标准部署方式：

  • 文章目录：source/_posts/
  • 站点配置：_config.yml
  • 输出目录：public/
  • 预览命令：npm run server
  • 构建命令：npm run build
  • 部署命令：npm run deploy

这条链路最终会把生成结果发布到 GitHub Pages。

2. 本地链路

本地链路只服务于私有内容：

  • 私有文章目录：local-only/_posts/
  • 本地覆盖配置：_config.local.yml
  • 临时源目录：.local-source/
  • 本地输出目录：public-local/
  • 构建命令：npm run build:local
  • 预览命令：npm run server:local

它不会触发 GitHub Pages 部署，也不会把私有文章写进公网输出目录。

这套双链路是怎么搭起来的

第一步：保留 Hexo 默认公网链路

这一部分其实不需要额外发明新机制，只保留 Hexo 的常规做法：

  • _config.yml 继续把 source_dir 指向 source
  • public_dir 继续是 public
  • deploy 继续指向 GitHub Pages 对应仓库

也就是说，公网链路本身尽量不要复杂化，后续维护最稳。

第二步：新增本地覆盖配置 _config.local.yml

  • public_dir: public-local 表示本地生成结果不再覆盖公网的 public/

  • source/_posts/ 里放准备公开发布的文章
  • local-only/_posts/ 里放私有文章

后面只要坚持这个约定，基本不会混。

第四步：用脚本生成 .local-source/

本地链路不是直接让 Hexo 去读 local-only/，而是先用脚本把本地专用源目录准备好，再交给 Hexo 生成。

当前项目里的脚本是 tools/prepare-local-source.ps1，它的逻辑是：

  1. 先清掉旧的 .local-source/
  2. 把 source/ 里除 _posts 以外的页面和静态资源复制进去
  3. 再把 local-only/ 的内容合并进去

这样得到的 .local-source/ 具备两个特点：

  • 保留了站点的页面、导航、图片、样式和其他资源
  • 文章部分只来自 local-only/_posts/

这就是“本地站只显示私有文章，不混公网文章”的关键。

这样做的好处是：

  • 不容易和默认公网预览端口混掉
  • 浏览器里一眼就能区分 4000 和 4001
  • 本地链路看到的就是实际生成后的 public-local/，结果更直接

场景 3：同时维护两边内容

这是最常见的情况。我的建议是把规则固定得非常机械：

  • 要公开的文章，一律写进 source/_posts/
  • 不公开的文章，一律写进 local-only/_posts/
  • 看公网效果，用 npm run server
  • 看私有专题，用 npm run server:local
  • 真正发站点时，只执行 npm run deploy

只要这几个动作不混，就不会把私有内容误发到线上。

我觉得最值得保留的几个约束

这套方案真正稳定，不是因为脚本多，而是因为边界清楚：

1. 公网链路尽量保持原生

公网那一套越接近 Hexo 默认行为，后续越不容易出问题。部署本身已经够关键，没必要在这条链上叠太多定制逻辑。

2. 本地链路一定要有独立输出目录

如果本地也写进 public/，就很容易和公网结果相互覆盖。public-local/ 这个隔离非常值。

3. 私有内容不要放进 source/_posts/

这一条看起来像废话，但实际上是最重要的纪律。目录边界一旦松掉，后面再靠命令补救会越来越累。

4. 本地服务和公网预览最好分端口

当前项目里：

  • 公网预览通常是 4000
  • 本地私有预览固定是 4001

这样开两个窗口时很难看错。

总结

这套双链路最核心的价值，不是“命令变多了”，而是把公开内容和私有内容真正隔离开了。

公网链路继续负责稳定发布，私有链路负责安心写作和专题整理。只要目录放对、命令用对，后续维护其实比把所有内容都混在一起更轻松，也更不容易出事故。
