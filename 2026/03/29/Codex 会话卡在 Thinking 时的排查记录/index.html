Codex App 卡在 Thinking 的处理办法

发表于2026-03-29|更新于2026-03-29|开发工具问题排查
|浏览量:

问题情况

Thinking 卡住，重启Codex、断开网络重联、甚至重启电脑后打开Codex 都还是 Thinking \ 正在思考。

先分两类：

  • 网络问题
  • App 处理数据问题

这一步的目的，是避免把链路问题误判为会话损坏，也避免把损坏会话误当成临时网络抖动。

一类：网络问题

典型表现

  • 新旧会话都可能异常
  • 不只影响单个会话
  • 重试后可能恢复
  • 切换网络、代理、provider 后现象变化明显

判断重点

优先看下面几件事：

  1. 新建会话是否也卡住
  2. 同时段其他会话是否正常
  3. 切网络后现象是否变化
  4. 切 provider 或 model 后是否变化
  5. CLI 与桌面端是否同时异常

处理顺序

  1. 先新建会话验证是否为全局问题
  2. 再检查当前网络、代理、中转和 provider
  3. 再切换网络环境复测
  4. 再比较不同 model / provider 的表现
  5. 最后才考虑是否存在本地状态异常

如果新旧会话一起不稳定，通常先按网络问题处理，不先动 session 文件。

二类：App 处理数据问题

典型表现

  • 新会话正常，只有某个旧会话异常
  • 问题集中在单个 session
  • 某次异常退出后开始出现
  • CLI 和桌面端对同一会话表现不一致

判断重点

需要记录的信息包括：

  • 问题会话标题
  • 问题出现时间
  • 对应 session id
  • 当前 provider / model
  • 本地会话文件修改时间
  • 新会话是否正常
  • CLI 是否能 resume 该会话

处理顺序

  1. 确认是否只有单个会话异常
  2. 确认新会话是否正常
  3. 尝试 CLI 恢复同一会话
  4. 定位对应 session 文件
  5. 先备份，再检查最后一段记录
  6. 在最小改动原则下处理该会话文件

如果前面的历史内容仍然完整，优先保留已有会话内容，再考虑如何继续当前对话。

文件操作原则

如果已经判断为 App 处理数据问题，处理本地会话文件时遵守下面几条：

  • 先备份原文件
  • 一次只改一个点
  • 每次改动后立即验证
  • 先做最小改动，不直接整段清空
  • 优先保住历史内容

结论

Thinking 卡住时，先判断它更像：

  • 网络问题
  • App 处理数据问题

前者先查链路，后者再查 session 数据。
